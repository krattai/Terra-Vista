How do I...?



NB: This is a draft. Any revisions can be posted to both me and the village. These guidelines have been drafted from methods we have agreed to as a community. If I have missed any, please let me know.

generalDevelopment

  • Terra Vista development is currently being done via strict VRML 1.0c.
  • Extension nodes that are VRML 1.0c compliant are acceptable for use.
  • Nodes that are browser specific and not part of VRML 1.0c (aside from VRML 1.0c extension nodes) are not encouraged.
  • Dynamic CGI wrls are acceptable to the extent that they should not limit the user or restrict surrounding environments/wrls. The use of CGI is still under discussion within the Terra Vista project.
  • Pre-test all development as best you can before releasing them within Terra Vista to help ensure that any user will be able to function within the environment.
  • Navigation within the community is part of the function that is being developed. The method is entirely up to the developer. The only recommendation is that when developing, try to include some device to allow for navigation.
  • It is requested that major reference to the Terra Vista project be ponted to the HTML and VRML home pages for the project. This does not disallow alternate "back-door" links.

    • zoneDevelopment

  • Navigation between zones is important to help establish a sense of connectedness within these virtual environments.
  • When connecting zones together, be sure to contact the perspective link zone builder to help establish recipricol links. If I can get to you, I should be able to return back from where I came via some device/method.
  • It is requested that all zones have a method to return the user to either the VRML or the HTML Terra Vista home pages.
  • If you choose to remove a link, for whatever reason, please inform the respective zoneBuilder of your intentions prior to the removal, so they may have an opportunity to update their link. Alternately, if you change the URL name that has a link attached to it, be sure to contact the link provider of the change.
  • Somewhere within the zone, possibly at the entrance to the zone, please include a mail device so a user may contact you with opinions and/or suggestions.

    • segmentDevelopment

  • Navigation between segments within a zone is important to maintain a sense of connectedness within the zone.
  • Try to maintain a connectedness that relieves a user from encountering "404 errors" or the equivalent.
  • Maintain a segment size and design that does not over-extend the load and parse time for that segment. Remember, the user has a stop button and may not give a kind review if this happens. A general rule of thumb is if it takes longer than you like to wait, then it's too big.

    • homeDevelopment

  • As a homeBuilder, one is limited only to the extent of the zoneBuilder's regulations. Please be sure to know the rules laid out by your prospective zoneBuilder.
  • Homes can be located anywhere on the internet and will be inlined into the zones.
  • In order to maintain a quick load and parse, it is recommended that the home only be an external shell file of the home. Also, it is recommended that you make extensive use of primitives wherever possible. A high usage of indexFaceSets and/or a large number of plygons is not recommended for the sake of speed.
  • The use of appropiate LOD nodes is recommended.
  • Any home "innards" should be WWWAnchored from some portal or transport device defined within the external shell.
  • What you have on the inside is entirely up to you. If the inside is browser specific or a large file, it is recommended that some indication is included in the external shell home suggesting that.
  • A home should include a mail box of some sort.
  • If you have an inside to your home, please be sure to establish a link back to the original segment from where the user came.
